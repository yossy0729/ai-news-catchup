## 1. 目的・ペルソナ

- ✅ **目的**: AIコンサルタント／リサーチャーが、**毎日5分**でAIの主要動向を「一次情報ベース」で把握できる。
- ✅ **主ペルソナ**: AIを業務で扱うコンサル・エンジニア・リサーチャー。二次まとめではなく原典で裏取りしたい人。
- 🟨 **コア価値**: 「速さ（5分）」×「信頼（一次情報）」×「漏れの少なさ（横断カテゴリ）」の同時達成。
- ✅ **【決定 2026-06-20】主軸 = AIコンサル/FDE。研究者は副**（完全には切らない）。
  ティッカーは consult/FDE レーンを主役に据える。軸をブレさせない。

## 2. スコープ

### やる（In）
- ✅ 一次情報ソースの自動巡回 → 候補化 → レビュー前処理 → ダッシュボード反映 → 検証、の日次パイプライン。
- ✅ カテゴリ別ニュース表示（各カテゴリ最大3件）、重要ニュースのスコア順表示、公式ベンダー動向、AI速報ティッカー。
- ✅ キーワード検索（保存済みニュース＋許可済み公式ソースの検索）。
- ✅ 任意のLLM要約（APIキーがある時のみ。無くてもルールベースで動作）。

### やらない（Out / 現時点）
- ✅ 二次メディア・SNS・ニュースアグリゲータ・非公式リポジトリを一次情報として採用しない。
- ✅ 有料記事・ログイン必須・クロール回避突破が必要なページの取得。
- 🟨 ユーザー認証・個人別パーソナライズ（公開フェーズで検討）。

## 3. 機能要件（MoSCoW）

### Must（必須）
- ✅ M1. 日次更新を1コマンド（`daily-update.js`）で実行でき、ドライランを持つ。
- ✅ M2. 当日更新が無いカテゴリは「前回分」または空で表示し、新着に `New` を付ける。
- ✅ M3. 各記事カードは要点・影響を示し、クリックで一次情報の原典に遷移する。
- ✅ M4. APIキー未設定でもアプリ全体が動作する（LLMはスキップ）。
- ✅ M5. データ検証（`validate-data.js`）が日次フローに含まれる。

### Should（推奨）
- 🟨 S1. 重要度スコアの質を上げる（§5）。
- 🟨 S2. 情報源の取得失敗を検知・可視化する（どのソースが壊れたか）。
- 🟨 S3. 一次情報が薄い日のフォールバック規則を明文化する（§4）。

### Could（任意）
- C1. ユーザーが「役立った/不要」をフィードバックし、ランキング改善に使う。
- C2. 管理者による除外・固定表示・再要約の編集機能（READMEの未決事項）。

### Won't（今回はやらない）
- W1. リアルタイム更新（日次バッチで十分）。
- W2. コメント・SNS的な双方向機能。

## 3.5 提案機能（新ウィジェット）★検討中

ニュースカードとは別の、プロ向け常設ウィジェット。いずれも「変化が遅く正確性が命」のため
自動スクレイピングではなく**キュレーション運用（出典＋時点を明記した小さなJSON）**を前提とする（§4）。

### P1. SOTAウォッチ（最新性能の早見） — Should
- 目的: 「今この課題の最高性能は何か」を一目で。
- 表示項目: **タスク(ベンチマーク/データセット) / 指標 / トップモデル / スコア / 時点 / 出典リンク**。
  - 例: `物体検出 (COCO test-dev) / box AP / Co-DETR / 66.0 / 2026-06 / [PwC]`
- 原則: データセットと指標を必ず併記（数値単独は無意味）。「◯時点」と出典を必須化。
- データ: 高価値タスク5〜10個に厳選した `sota.json`（手動キュレーション、出典は PwC / HF Leaderboard 等）。
- ✅ **【決定 2026-06-20】全網羅は避け、ドメインタブで短く保つ**（すべて/言語・推論/画像・映像/音声/
  マルチモーダル/コード・エージェント）。各行に出典リンク（全部見たい人はリーダーボードへ誘導）。
- ✅ **【決定 2026-06-21】値と出典のハイブリッド**: 信頼できる公式リーダーボードから取れる値は実値
  (verified)、取りにくい/定番出典消失のものは出典サイト名＋リンクのみ(score=「—」)。誤値は載せない。
- ✅ **【決定 2026-06-21】推移は「2行兼用」表示**: 列を増やさず、上段=今回SOTA(緑背景・太字)/下段=前回
  SOTA(薄灰)。改善幅は「◯pt / 約N年ぶりに更新」。**同一ベンチ・同一指標(prevComparable)のときだけ**
  数値差分を出し、ハーネス/基準が変わった場合は「基準変更のため単純比較不可」と明示（誤比較回避）。
  指標の向き(higherIsBetter)を考慮。時点は月日まで（前回も公式リリース日を調べて月日で記載）。
- 🟡 **将来(CODEX) SOTA履歴の自動スナップショット**: 週1〜月1でリーダーボードを確認し、**1位モデルが
  交代した時だけ**現在値→前回値へ退避して新値を記入。これで「前回→今回」の推移が自然に蓄積する
  （毎日は不要・コスト無駄）。取得は403/JS対策でブラウザ拡張方式か公式APIを使う。順位交代の速報化も。
- ❓**[要決定]** スコアの更新頻度（週1/月1）と維持担当（自動化までは手動更新）。

### P2. 研究の注目株（HF Daily Papers の再構成） — Could
- 現状: HF Daily Papers は**コミュニティ投票(upvote)順**＝人気シグナルであり、重要度保証ではない。
- 課題: 生の英語abstractは日本語ペルソナに低価値。
- 見せ方: **日本語1〜2行で「何が新しい/なぜ注目/実務インパクト」** ＋ 原題 ＋ 投票数(根拠明示) ＋ arXiv原典。
  - ラベルに「※人気指標であり重要度保証ではない」を明示（誠実さ）。
- 方針: 良い日本語要約を作れる時のみ表示価値が高い（LLM要約が活きる場所）。作れないなら出さない判断も可。
- ❓**[要決定]** 残す/縮小/廃止。LLM日本語要約を必須前提とするか。

### P3. モデル価格早見表 — Should
- 目的: 「このモデルの出力は100万トークンあたりいくらか」を即把握（コンサルの最大関心事）。
- 表示項目: **ベンダー / モデル / 入力$/1M / 出力$/1M / コンテキスト長 / 時点 / 公式pricingリンク**。
  - 入力と出力を分けて表示（出力が高いことが多く誤解を防ぐ）。
  - ハイライト: 「出力最安」「今月の価格改定」。
- 派生: **価格改定の検知をニュース化**（値下げ/値上げは重要シグナル）。
- データ: 手動キュレーションの `pricing.json`（時点＋出典必須。価格は月単位変化で運用可能）。
- ❓**[要決定]** 掲載ベンダー/モデルの初期セット。通貨（USD固定/円換算併記）。

## 4. データ・情報源ポリシー

- ✅ **一次情報の定義**: 官公庁・規制当局・企業ニュースルーム・公式ブログ・研究機関・論文インデックス・公式GitHub・標準化団体（`sources.json` の `sourceType`）。
- ✅ **取得方式**: `html` / `rss` / `api` / `github` / `manual_review`（`fetchMethod`）。
- 🟨 **自動収集 vs キュレーションの使い分け（重要原則）**:
  - 毎日変わる情報（ニュース）→ 自動収集が向く。
  - 変化が遅く正確性が命の情報（SOTA数値・モデル価格）→ **手動キュレーション**で持つ（出典＋時点を明記）。
  - 理由: 数値の自動スクレイピングは誤値を出して信頼を失うリスクが高い（§5の検証根拠と同根）。
- ✅ **候補のライフサイクル**: `candidate → accepted / rejected / expired`。14日未検出で expired、30日超で削除対象。【決定 2026-06-20】現状維持。
- ✅ **【決定 2026-06-20】一次情報と網羅の両立は「場所で役割分担」（折衷案C）**:
  - **メイン枠（重要ニュース・カテゴリ別）＝一次情報の聖域**。二次情報を混ぜない。
  - **速報ティッカー＝動向把握用**。二次情報(メディア速報)も"参考"ラベル付きで許容。
  - これにより、信頼性（理念）と毎日の充実感（網羅）を構造で両立する。
- ✅ **【決定 2026-06-20】RSS/API が使えるソースは優先的にそちらへ寄せる。HTMLスクレイピングは最終手段**（壊れにくさ優先）。

## 5. 重要度スコアの仕様 ★最重要

> このアプリの価値の核心。現状はルールベース（信頼度・ソース優先度・鮮度・カテゴリ・AIキーワード加点）。READMEも「LLB補助で再評価」を将来課題としている。

- 🟨 **現状の加点要素**: 一次情報の信頼度 / ソース優先度 / 鮮度 / カテゴリ / AI関連キーワード。
- ✅ **【決定 2026-06-20】「重要」の定義 = 次の5軸**（ペルソナ: AIコンサル/リサーチャー）:
  1. **技術的ブレークスルー**: 新SOTA・新能力・新アーキテクチャ。
  2. **ビジネス/調達インパクト**: 価格改定・ライセンス変更・新API（提案や見積もりに直結）。
  3. **規制・法判断**: 著作権判決・規制発効・ガバナンス（クライアント助言に直結）。
  4. **実装・運用に効く**: RAG/エージェント/MCP・運用事例（現場で即使える）。
  5. **社会的インパクト**: 大型投資・M&A・重大インシデント（業界地図が変わる）。
  - 既存ティッカー設計(core/FDE/research/consult)と整合。これを公式の評価軸とする。
- ✅ **【決定 2026-06-20】算定方式 = ハイブリッド(案C)**:
  - ルールベースで粗く候補を絞る（無料）→ **上位10〜15件のみ** LLM が上記5軸で再評価・並べ替え。
  - 1日数円〜数十円に収まり、質を大きく上げられる。APIキー未設定時はルールベースのみに自動フォールバック。
- ✅ **【決定 2026-06-20】"検証済み"という表現は使わない**。自動判定で「検証済み」は誇大なため、
  事実ベースの「一次情報」「公式」等の表現に統一する（画面文言の見直し対象）。

## 6. 非機能要件

- ✅ **コスト**: LLM課金は「1日1回の生成」に限定。閲覧は静的JSONで無料。要約は安価モデル(mini/nano)を既定。
- 🟨 **コスト上限**: OpenAI ダッシュボードで Usage limits と請求アラートを設定（保険）。
- ✅ **秘匿情報**: APIキーはサーバー側のみ。フロントに出さない。`.env` は Git 管理外。
- 🟨 **セキュリティ（公開時 Must 化）**:
  - `/api/update` は認証なしで誰でも叩ける → 公開前に **認証＋レート制限**（連打で課金されるため）。
  - 静的配信のパストラバーサル対策済み（実装済み）。
- 🟨 **性能**: ローカル単一ユーザー前提。同期I/Oで可。多人数化時に非同期化を検討。
- 🟨 **可用性**: Windowsタスクスケジューラで毎朝自動実行。失敗はログ（`logs/`）に残す。
