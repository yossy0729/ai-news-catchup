# AI News Catchup 要件定義書

最終更新: 2026-07-02

## 0. 文書の位置づけ

この文書は AI News Catchup の「何を満たすべきか」を定義する正本です。
実装手順や運用コマンドは `README.md`、作業経緯や引き継ぎは `HANDOFF.md` に分けます。

ステータス表記:

- `[実装済み]`: 現行コードとデータで満たしている
- `[一部実装]`: 基本は動くが品質・自動化・運用上の課題が残る
- `[未実装]`: 要件として残すが現行スコープ外

## 1. 目的・ペルソナ

- `[実装済み]` AIコンサルタント/FDE/リサーチャーが、毎日5分でAIの主要動向を確認できる。
- `[実装済み]` 主ペルソナはAIを業務で扱うコンサルタント、FDE、エンジニア。研究者は副次ペルソナとする。
- `[実装済み]` 価値の軸は「速報性」「一次情報への到達」「横断カテゴリの漏れにくさ」。
- `[実装済み]` メイン枠では一次情報を優先し、速報枠では二次メディアを参考情報として扱う。

## 2. 現行プロダクトの範囲

### In Scope

- `[実装済み]` 静的ダッシュボードを GitHub Pages で公開する。
- `[実装済み]` ローカルサーバー `serve.js` で静的配信、公式ソース検索、候補保存、手動更新を提供する。
- `[実装済み]` GitHub Actions で日次データ更新と Pages デプロイを実行する。
- `[実装済み]` ニュース、メディア速報、公式ベンダー動向、AIシグナル、モデル価格、SOTA指標をJSONで管理する。
- `[実装済み]` OpenAI APIキーがある場合のみ、日本語タイトル翻訳と要約補助を実行する。キーが無い場合はルールベースで継続する。
- `[実装済み]` ローカル環境では「今すぐ取得」ボタンから日次更新を手動実行できる。

### Out of Scope

- `[未実装]` リアルタイム更新。日次バッチで十分とする。
- `[未実装]` コメント、SNS投稿、ユーザー同士の双方向機能。
- `[未実装]` 個人別レコメンド、ログイン、ユーザーごとの既読管理。
- `[実装済み]` 有料記事、ログイン必須ページ、クロール回避を突破する取得は行わない。
- `[実装済み]` 二次メディアやニュースアグリゲータを一次情報として扱わない。

## 3. 画面要件

### 共通

- `[実装済み]` 画面は `今日の動向`、`カテゴリ別`、`公式動向`、`指標・ベンチ` の4タブで構成する。
- `[実装済み]` ヘッダーに基準日、速報件数、New件数、一次情報件数を表示する。
- `[実装済み]` キーワード検索は保存済みの速報・一次情報を横断して絞り込む。
- `[実装済み]` ローカルサーバー（localhost / 127.0.0.1）で開いた場合のみ、公式ソース検索、候補保存、手動更新を操作可能にする。GitHub Pages や `file://` 直開きなど、サーバーAPIが使えない環境では閲覧専用にする。

### 今日の動向

- `[実装済み]` AIメディア速報を表示する。24時間以内の新着を優先し、重要カテゴリが空になる場合は直近72時間の注目ニュースで補完する。
- `[実装済み]` メディア速報は元記事URLと公開日時を確認できる固定ソースを対象にする。
- `[実装済み]` 24時間以内を `fresh`、24から72時間を `recent` として区別し、空カテゴリはrecentで補完できる。
- `[実装済み]` AI FLASH ティッカーは `主要`、`FDE`、`研究・モデル`、`規制・コンサル` のレーンを持つ。

### カテゴリ別

- `[実装済み]` 一次情報ニュースを `全分野`、`モデル・研究`、`産業導入`、`規制・ガバナンス`、`セキュリティ`、`インフラ・基盤` で束ねて表示する。
- `[実装済み]` 元データは10カテゴリを維持し、画面上では分野軸に再編する。
- `[実装済み]` 各分野は日付降順で最大4件を表示する。
- `[実装済み]` 国内/海外/横断の違いはカード上の地域バッジとカテゴリ情報で示す。
- `[実装済み]` 一次情報は毎日更新されないため、数日から数週間前の項目が並ぶことを許容する。

### 公式動向

- `[実装済み]` AIベンダー/GAFAM/主要国内AI関連組織の公式発表をベンダー別に表示する。
- `[実装済み]` ベンダーカードの色はブランド識別色とし、ニュース内容のトーンとは分離する。
- `[実装済み]` 公式動向にも24時間以内の更新はNew表示を付ける。

### 指標・ベンチ

- `[実装済み]` モデル価格早見表を表示する。項目はベンダー、モデル、入力単価、キャッシュ入力単価、出力単価、出力倍率、コンテキスト長、出典リンク、確認状態。
- `[実装済み]` SOTAウォッチを表示する。項目は分野、ベンチマーク、指標、トップモデル、スコア、時点、出典リンク。
- `[実装済み]` SOTAは注目タブを既定表示にし、全分野検索とドメインタブを提供する。
- `[実装済み]` SOTAの前回値がある場合は、今回値と前回値を2行で表示する。
- `[実装済み]` 指標の向きは上向き/下向きで表示し、数値が無い分野は「データ未登録」としてリンクのみ表示する。

## 4. データ要件

### データファイル

- `[実装済み]` `data/news.json`: 一次情報ニュース。10カテゴリを保持する。
- `[実装済み]` `data/media-news.json`: 直近AIメディア速報。
- `[実装済み]` `data/official-news.json`: 公式ベンダー動向。
- `[実装済み]` `data/ai-signals.json`: ティッカー向けAIシグナル。
- `[実装済み]` `data/pricing.json`: モデル価格早見表。
- `[実装済み]` `data/sota.json`: SOTAウォッチ。
- `[実装済み]` `data/sources.json`: 一次情報ソース定義。
- `[実装済み]` `data/candidates.json`: 収集候補キュー。
- `[実装済み]` `data/review.json`: レビュー前処理済み候補。
- `[実装済み]` `data/health.json`: 日次更新全体の成功/警告/失敗、件数、実行ステップを記録する。
- `[実装済み]` `data/source-health.json`: ソース別の取得状態、取得件数、失敗理由を記録する。

### 情報源ポリシー

- `[実装済み]` 一次情報の定義は、官公庁、規制当局、企業ニュースルーム、公式ブログ、研究機関、論文インデックス、公式GitHub、標準化団体とする。
- `[実装済み]` `sourceType` と `fetchMethod` は `data/sources.json` で管理し、`validate-data.js` で妥当性を検証する。
- `[実装済み]` RSS/APIが使えるソースを優先し、HTML取得は最終手段とする。
- `[実装済み]` Google News RSS の発見リンクは、元記事URLと公開日の検証なしでは表示データに採用しない。
- `[実装済み]` メディア速報と公式動向は `dateVerification.status === "verified"` を必須とする。
- `[実装済み]` 公開カードの見出し・要約は、元記事タイトルや本文表現のコピーではなく、AIニュース把握用の独自見出しと独自要約を表示する。日本語記事も対象とし、海外記事は90〜150字程度・2文以内の日本語要約にする。

### 候補ライフサイクル

- `[実装済み]` 候補の状態は `candidate`、`accepted`、`rejected`、`expired` のいずれかとする。
- `[実装済み]` `candidate` は14日以上再検出されない場合に `expired` とする。30日を超えた `expired` は削除対象とする。期限切れ処理（`prune-candidates.js`）は日次更新フローで候補収集の直後に自動実行する。
- `[実装済み]` 同じURLを再保存しても、`accepted` や `rejected` を勝手に `candidate` へ戻さない。
- `[実装済み]` `candidates.json` と `review.json` はローカル生成物としてGit管理外とする。これらが存在しないクリーンな環境でも、日次更新・データ検証は空キューとして継続する。
- `[一部実装]` GitHub Actions 経路では候補・レビューのキューが毎回空から再構築される（コミット対象外のため）。表示の重複は `news.json` のURL重複チェックで防ぐ。候補の採否・期限切れの履歴が意味を持つのはローカル運用のみ。

## 5. 更新パイプライン

- `[実装済み]` 日次更新は `node scripts/daily-update.js` で実行できる。
- `[実装済み]` `--dry-run` でデータを書き換えずに確認できる。
- `[実装済み]` 標準フローは次の順序で実行する。
  1. 公式動向収集
  2. AIシグナル収集
  3. メディア速報収集
  4. 一次情報候補収集
  5. 候補キューの期限切れ処理
  6. 任意の日本語タイトル翻訳
  7. レビュー前処理
  8. 任意のLLM要約
  9. ニュース反映
  10. 任意の既存ニュース要約補完
  11. SOTA収集
  12. データ検証
- `[実装済み]` SOTA収集は外部API依存のため、失敗してもニュース更新全体は止めない。
- `[実装済み]` 日次更新ログは `logs/daily-update-YYYY-MM-DD.log` に保存する。
- `[実装済み]` GitHub Actions はスケジュール実行と手動実行でデータ生成し、Pagesへ公開する。
- `[一部実装]` ローカルのWindowsタスクスケジューラ実行は補助経路とする。登録状態は環境依存のため、正本の運用経路はGitHub Actionsとする。

## 6. スコアリング・要約要件

- `[実装済み]` 重要度スコアはルールベースを基本とする。
- `[実装済み]` 加点軸はソース信頼度、ソース優先度、鮮度、カテゴリ、AI関連キーワードとする。
- `[実装済み]` 一次情報ニュースには `importance`、`importanceLabel`、`importanceReason` を持たせ、技術/事業/規制/実装/市場の5軸で「なぜ重要か」を記録する。
- `[実装済み]` カテゴリ別表示と一次情報ティッカーは、重要度を主軸にし、同点時に新しさで並べる。
- `[一部実装]` LLMが利用できる場合は、日本語要約・タイトル翻訳の品質向上に使う。
- `[未実装]` LLMによる重要度ランキング再評価は将来課題とする。
- `[実装済み]` 「検証済み」という表現は自動判定では使わず、「一次情報」「公式」「確認済」など根拠に応じた表現にする。
- `[実装済み]` カード要約は日本語要約がある場合に優先し、無い場合は既存要約へフォールバックする。

重要度の判断軸:

1. 技術的ブレークスルー: 新SOTA、新能力、新アーキテクチャ。
2. ビジネス/調達インパクト: 価格改定、ライセンス変更、新API。
3. 規制・法判断: 著作権判決、規制発効、ガバナンス。
4. 実装・運用に効く情報: RAG、エージェント、MCP、導入事例。
5. 社会的インパクト: 大型投資、M&A、重大インシデント。

## 7. SOTA・価格データ要件

### SOTA

- `[実装済み]` Papers with Code API を主ソースとして全研究分野のSOTA候補を収集する。
- `[実装済み]` PwCで弱い分野は公式リーダーボードで上書きできる。
- `[実装済み]` 代表ベンチは `data/sota-datasets.json` で手動指定できる。
- `[実装済み]` 日本語ラベル・検索語は `data/sota-labels.json` で補う。
- `[実装済み]` 1位モデルまたはスコアが変わった場合、現在値を `prev*` に退避して履歴表示に使う。
- `[実装済み]` SOTA表の日付は `記録日 / 確認日` として表示し、画面上でも `記録:` / `確認:` を付けて意味を明示する。PwCなど元データ側に日付があるものは記録日を表示し、LMArena / Open ASR のように新SOTA日が取れない公式リーダーボードは初回確認日を維持して表示する。
- `[一部実装]` すべての分野で数値が取れるわけではない。数値が無い分野はリンクのみ掲載する。

### 価格

- `[実装済み]` 価格は `data/pricing.json` の手動キュレーションを正とする。
- `[実装済み]` 単価はUSD/100万トークンを基本単位とする。
- `[実装済み]` `verified` が false の価格は参考扱いにする。
- `[未実装]` 価格改定の自動検知とニュース化は将来課題とする。

## 8. 非機能要件

- `[実装済み]` フロントエンドは静的HTML/CSS/JSで動作し、閲覧時にサーバーコストを発生させない。
- `[実装済み]` APIキーはサーバー側またはGitHub Actions Secretsで扱い、フロントへ露出しない。
- `[実装済み]` APIキー未設定でも日次更新と閲覧を継続できる。
- `[実装済み]` 静的配信はパストラバーサルを防ぐ。
- `[実装済み]` ローカルサーバーは不正なリクエスト（不正なパーセントエンコーディング等）で停止せず、4xxを返して稼働を継続する。
- `[実装済み]` 表示用データの一部（health.json等の任意データ）が欠けていても、画面全体を読み込み失敗にしない。
- `[実装済み]` 外部HTML取得はサイズ上限を設ける。
- `[一部実装]` `/api/update`、`/api/search`、`/api/candidates` はローカル利用前提。公開サーバー化する場合は認証、CSRF対策、レート制限が必須。
- `[実装済み]` データ取得失敗はログ、各データの `errors`、`data/source-health.json` に残す。
- `[一部実装]` 画面上では更新全体の正常/警告/失敗を表示する。ソース健全性の詳細ダッシュボードは未実装。
- `[一部実装]` テストはデータ検証とJS構文チェックが中心。UI自動テストは未整備。

## 9. 検証・受け入れ基準

- `[実装済み]` `node scripts/validate-data.js` が成功すること。クリーンなclone直後（`candidates.json` / `review.json` 未生成）でも成功すること。
- `[実装済み]` `app.js`、`serve.js`、`scripts/*.js` が `node --check` を通ること。
- `[実装済み]` `data/sources.json` のカテゴリ、ソース種別、取得方式が定義済み集合と一致すること。
- `[実装済み]` `candidates.json` と `review.json` に重複URLが無いこと。
- `[実装済み]` `media-news.json` と `official-news.json` は元記事URL、公開日検証、重複URLチェックを通ること。
- `[実装済み]` `ai-signals.json` はID、レーン、タグ、本文、URL、日付を持つこと。
- `[一部実装]` 公開前の目視確認では、4タブが読み込めること、検索が動くこと、空データ時の表示が破綻しないことを確認する。

## 10. 残課題・ロードマップ

### Phase 1: 個人MVP

- `[一部実装]` 収集対象ソースとカテゴリ設計を継続的に調整する。
- `[一部実装]` LLM要約の品質を確認し、課金上限を運用で管理する。
- `[一部実装]` 研究の注目株（HF Daily Papers系）を残すか、AIシグナルへ吸収するか決める。

### Phase 2: 少人数利用

- `[未実装]` ローカルAPIを公開サーバー化する場合は、認証、レート制限、CSRF対策、実行キューを入れる。
- `[未実装]` ソース別の取得成功率、失敗理由、最終成功日時を画面で見られるようにする。
- `[未実装]` 管理者による除外、固定表示、再要約、候補レビューUIを追加する。

### Phase 3: 公開プロダクト化

- `[未実装]` UI自動テスト、日次更新の監視、通知、障害時リカバリを整備する。
- `[未実装]` LLMによる重要度再評価を導入し、ルールベースとのハイブリッドランキングにする。
- `[未実装]` 価格改定やSOTA交代の重要イベントをニュース枠へ昇格できるようにする。
